# RL文档简介

在这个高级主题中，我们解决以下问题： **我们应该如何监控和跟踪我们在现实世界中训练的强大的强化学习代理
与人类交互？**

随着机器学习系统对现代生活的影响越来越大， **对这些系统的文档的呼声越来越高**.

此类文档可以涵盖使用的培训数据等方面——存储数据的位置、收集时间、参与人员等。
— 或模型优化框架 — 架构、评估指标、相关论文等 — 等等。

如今，模型卡和数据表变得越来越可用。例如，在 Hub 上
(查看文档 [这里](https://huggingface.co/docs/hub/model-cards)).

如果您点击 [Hub 上的热门型号](https://huggingface.co/models)，您可以了解它的创作过程。

这些模型和数据特定日志旨在在创建模型或数据集时完成，而当这些模型将来构建到不断发展的系统中时，它们不会更新。
​
## 激励奖励报告

强化学习系统从根本上设计为基于奖励和时间的测量进行优化。
虽然奖励函数的概念可以很好地映射到监督学习的许多易于理解的领域 (通过损失函数),
对机器学习系统如何随时间演变的理解是有限的。

为此，作者介绍 [*强化学习的奖励报告*](https://www.notion.so/Brief-introduction-to-RL-documentation-b8cbda5a6f5242338e0756e6bef72af4) (简洁的命名旨在反映流行论文 *用于模型报告的模型卡* 和 *数据集的数据表*).
目标是提出一种专注于以下内容的文档类型： **奖励的人为因素** 和 **时变反馈系统**.

建立在文档框架的基础上 [模型卡](https://arxiv.org/abs/1810.03993) 和 [数据表](https://arxiv.org/abs/1803.09010) 由米切尔等人提出。和 Gebru 等人，我们认为人工智能系统需要奖励报告。

**奖励报告** 是拟议的 RL 部署的动态文档，用于划分设计选择。

然而，关于该框架对不同 RL 应用程序的适用性、系统可解释性的障碍、
以及部署的监督机器学习系统与 RL 中使用的顺序决策之间的共振。

至少，奖励报告为 RL 从业者提供了一个机会来思考这些问题并开始决定如何在实践中解决这些问题。
​
## 通过文档捕获时间行为

专为 RL 和反馈驱动的 ML 系统设计的文档的核心部分是 *变更日志*。变更日志更新信息
来自设计师 (改变训练参数、数据等) 以及用户注意到的变化 (有害行为、意外反应等).

更改日志附有更新触发器，鼓励监视这些影响。

## 贡献

一些最有影响力的 RL 驱动系统本质上是多利益相关者的，并且是在私营公司的闭门之后。
这些公司基本上没有监管，因此记录的负担就落在了公众身上。

如果您有兴趣做出贡献，我们正在为公共领域的流行机器学习系统构建奖励报告
记录在 [GitHub](https://github.com/RewardReports/reward-reports).
​
如需进一步阅读，您可以访问奖励报告 [纸](https://arxiv.org/abs/2204.10817)
或者看 [报告示例](https://github.com/RewardReports/reward-reports/tree/main/examples).
